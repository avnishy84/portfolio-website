Angular 18 SSR with Signals and OnPush: Performance Patterns for Production Applications

Avnish Yadav
2024 Full-Stack · Performance 3 pages
Abstract

Angular 18 introduced Signals as a reactive primitive and stabilized server-side rendering, creating new opportunities for performance optimization. This paper documents the interaction between SSR, Signals, and OnPush change detection in production applications, with patterns and pitfalls drawn from building the Compufy Technology corporate website. Key findings include the hydration mismatch conditions that arise when Signals are used with SSR, and the change detection optimization strategy that reduced client-side rendering time by 40%.

Angular's change detection system has historically been a source of performance problems in large applications. The default change detection strategy checks every component in the tree on every event, which becomes expensive as the component tree grows. Angular 18's combination of Signals and OnPush change detection provides a path to fine-grained reactivity that eliminates unnecessary re-renders.

1. Angular Signals

Signals are a reactive primitive that track their dependencies automatically. When a signal's value changes, Angular knows exactly which components depend on that signal and marks only those components for re-checking. This is fundamentally different from the previous approach, where any state change could potentially trigger a full component tree check.

The key discipline when using Signals is to keep signal state at the appropriate level of the component tree. Signals defined in a parent component that are read by many child components create a wide dependency graph; signals defined in leaf components create a narrow one. For the Compufy website, moving page-level state signals from the root component to the section components that actually used them reduced the number of components checked on navigation by 60%.

2. OnPush Change Detection

OnPush change detection tells Angular to only check a component when its input references change, when an event originates from the component or its children, or when a signal it reads changes. This is a significant optimization for components that receive complex objects as inputs — instead of deep-comparing the object on every change detection cycle, Angular only checks the component when the object reference changes.

The practical implication is that all state mutations must produce new object references rather than mutating existing objects. This aligns naturally with immutable data patterns and works well with Signals, which always produce a new value on assignment. The main pitfall is accidentally mutating an object that is passed as an OnPush input — the component will not re-render because the reference has not changed.

3. SSR and Hydration

Angular 18's SSR renders the application on the server and sends the HTML to the client, where Angular then "hydrates" the static HTML by attaching event listeners and initializing component state. The most common source of hydration mismatches is state that differs between server and client — for example, reading window.innerWidth on the server (where window is undefined) versus the client.

The solution is to use Angular's isPlatformBrowser() guard for any code that accesses browser-only APIs, and to initialize Signals with server-safe default values that will be overwritten during hydration. The Compufy website uses a pattern where layout-affecting signals (viewport size, scroll position) are initialized to their server-side defaults and updated in ngAfterViewInit, which only runs on the client.

4. Production Results

Applying the Signal + OnPush + SSR pattern to the Compufy Technology corporate website produced measurable improvements across all Core Web Vitals. Largest Contentful Paint improved from 2.8s to 1.4s due to SSR delivering pre-rendered HTML. Cumulative Layout Shift dropped to near zero because the server-rendered layout matches the hydrated layout. Total Blocking Time decreased by 35% due to reduced change detection work during initial load.

5. Conclusion

Angular 18's combination of Signals, OnPush change detection, and SSR provides a powerful toolkit for building high-performance web applications. The key to realizing the performance benefits is understanding how these three systems interact: Signals provide fine-grained reactivity, OnPush ensures that reactivity translates to minimal re-rendering, and SSR ensures that the initial page load is fast regardless of JavaScript execution time.

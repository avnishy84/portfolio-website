Creational Design Patterns in Java: A Practical Guide with Code Examples

Avnish Yadav
2025 Software Engineering · Design Patterns 3 pages
Abstract

Creational design patterns solve the problem of object creation — not just how to create objects, but how to create them in a flexible, reusable, and decoupled way. This paper explains the five canonical creational patterns (Singleton, Factory, Builder, Prototype, Abstract Factory) through practical Java code examples, real-world analogies, and clear guidance on when each pattern is the right tool for the job.

Creating objects with the new keyword is easy. Creating them in a way that is maintainable, testable, and open to change is harder. Creational patterns are the answer to that harder problem. Think of it like a car factory — you don't build each car by hand every time. You build a system that produces cars consistently and efficiently. That system is the creational pattern.

1. Singleton Pattern

Singleton ensures that a class has only one instance and provides a global access point to it. The classic use cases are database connections and loggers — resources that are expensive to create and should be shared across the application.

class Singleton {
    private static Singleton instance;

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {
            instance = new Singleton();
        }
        return instance;
    }
}

The constructor is private so no external code can call new Singleton(). The static getInstance() method creates the instance on first call and returns the same instance on every subsequent call. In modern Java, prefer using a DI container with singleton scope over implementing this pattern manually — it avoids the hidden global state problem.

2. Factory Pattern

Factory Pattern hides object creation behind a method. The caller asks for an object by type; the factory decides which concrete class to instantiate. This decouples the caller from the implementation — you can add new types without changing the calling code.

interface Shape { void draw(); }

class Circle implements Shape {
    public void draw() {
        System.out.println("Drawing Circle");
    }
}

class ShapeFactory {
    public Shape getShape(String type) {
        if (type.equals("circle")) return new Circle();
        return null;
    }
}

The caller never uses new Circle() directly — it calls factory.getShape("circle"). Adding a new shape (Triangle, Rectangle) only requires adding a new class and updating the factory — existing callers are untouched.

3. Builder Pattern

Builder constructs complex objects step by step. It is the right pattern when an object has many optional fields — the alternative (telescoping constructors) becomes unreadable fast.

class User {
    private String name;
    private int age;

    public static class Builder {
        private String name;
        private int age;

        public Builder setName(String name) {
            this.name = name; return this;
        }
        public Builder setAge(int age) {
            this.age = age; return this;
        }
        public User build() {
            User u = new User();
            u.name = this.name;
            u.age = this.age;
            return u;
        }
    }
}

// Usage:
User user = new User.Builder()
    .setName("Avnish")
    .setAge(25)
    .build();

Each setter returns this, enabling fluent method chaining. The build() call at the end validates the accumulated state and returns the final immutable object.

4. Prototype Pattern

Prototype creates new objects by cloning an existing one. Use this when object creation is expensive (e.g., loading from a database or network) and you need many similar objects — clone the expensive one and modify the copy.

class Prototype implements Cloneable {
    public String data;

    public Prototype clone()
            throws CloneNotSupportedException {
        return (Prototype) super.clone();
    }
}

// Usage:
Prototype original = new Prototype();
original.data = "original";
Prototype copy = original.clone();
copy.data = "copy"; // original unchanged

Java's Cloneable interface and super.clone() provide shallow cloning. For deep cloning (when the object contains references to other objects), implement a custom copy constructor or use serialization.

5. Abstract Factory Pattern

Abstract Factory produces families of related objects. The classic example is UI theming — a Dark theme factory produces dark buttons, dark dialogs, and dark inputs; a Light theme factory produces the light equivalents. The application code works with the factory interface and never knows which theme is active.

interface UIFactory {
    Button createButton();
    Dialog createDialog();
}

class DarkUIFactory implements UIFactory {
    public Button createButton() {
        return new DarkButton();
    }
    public Dialog createDialog() {
        return new DarkDialog();
    }
}

6. When to Use Each Pattern

Pattern Use When
SingletonOne shared instance needed (DB connection, logger)
FactoryType to create is determined at runtime
BuilderComplex object with many optional fields
PrototypeCloning is cheaper than creating from scratch
Abstract FactoryFamilies of related objects (themes, platforms)

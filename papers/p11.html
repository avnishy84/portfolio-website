Firebase as a Full-Stack Platform: Authentication, Firestore, and Hosting Patterns

Avnish Yadav
2024 Full-Stack · Database 3 pages
Abstract

Firebase provides a complete backend-as-a-service platform that enables small teams to build production-grade applications without managing server infrastructure. This paper documents practical patterns for Firebase Authentication, Firestore data modeling, security rules, and Hosting, drawn from three production applications: One Note (collaborative note-taking), Compufy Technology (corporate website with analytics), and CVio (AI-powered resume optimizer). Each application presented distinct challenges that illuminate different aspects of the Firebase platform.

Firebase's appeal for small teams and solo developers is its ability to eliminate the operational overhead of managing authentication, databases, and hosting infrastructure. However, Firebase's flexibility comes with footguns — data modeling decisions that seem reasonable at small scale become expensive at large scale, and security rules that seem restrictive enough can have subtle gaps.

1. Firebase Authentication

Firebase Authentication handles the complexity of OAuth flows, token management, and session persistence. The key architectural decision is how to model the relationship between Firebase Auth users and application-specific user data. The recommended pattern is to create a Firestore document in a users collection with the same ID as the Firebase Auth UID on first sign-in, and to store all application-specific user data there rather than in Firebase Auth's user profile.

This separation keeps Firebase Auth as a pure identity provider and Firestore as the source of truth for application data, making it straightforward to add fields to the user profile without touching the auth layer.

2. Firestore Data Modeling

Firestore's document-collection model requires a different approach to data modeling than relational databases. The key principle is to model data for how it will be queried, not for normalization. Firestore does not support joins, so data that is frequently queried together should be stored together, even if this means denormalization.

For One Note, the data model stores note metadata (title, last modified, collaborators) in a top-level notes collection and note content in a separate content subcollection. This allows the notes list to be loaded without fetching the full content of each note, reducing read costs and improving load time for users with many notes.

For CVio, the resume optimization history is stored as an array of optimization records within the user document rather than as a separate collection. This works because the array is bounded (users are limited to 50 optimization records) and is always queried as a unit. Storing it as a subcollection would require an additional query and increase read costs.

3. Security Rules

Firestore security rules are the primary defense against unauthorized data access. The most common mistake is writing rules that are correct for the happy path but have gaps for edge cases. The recommended approach is to write rules that explicitly deny access by default and grant access only for specific, well-defined conditions.

A critical pattern is validating incoming data in security rules, not just access control. Rules should verify that required fields are present, that field types are correct, and that field values are within acceptable ranges. This prevents malformed data from entering the database even if the client-side validation is bypassed.

4. Firebase Hosting

Firebase Hosting provides CDN-backed static hosting with SSL, custom domains, and deployment previews. For single-page applications, the key configuration is the rewrites rule that serves index.html for all routes, enabling client-side routing. For multi-page applications like this portfolio, the default behavior of serving the matching HTML file is correct.

Firebase Hosting's deployment preview feature — which creates a unique URL for each deployment — is particularly valuable for reviewing changes before promoting them to production. The firebase.json hosting configuration supports headers for cache control, allowing static assets to be cached aggressively while HTML files are served with no-cache headers to ensure users always get the latest version.

5. Cost Management

Firebase's free tier (Spark plan) is generous for development and small production applications, but costs can grow unexpectedly as usage scales. The primary cost drivers are Firestore reads and writes, Cloud Functions invocations, and Authentication monthly active users. Monitoring these metrics in the Firebase console and setting budget alerts is essential before launching a production application.
